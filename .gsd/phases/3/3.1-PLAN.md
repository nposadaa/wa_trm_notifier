# Plan 3.1: Multi-Recipient Support

## Objective
Pivot from group messaging (not supported by Meta API) to an individual broadcasting model. This involves creating a `recipients.json` file for managing multiple contacts and updating the orchestrator (`main.py`) to loop through them.

## Context
- .gsd/SPEC.md
- .gsd/phases/3/RESEARCH.md
- main.py
- .env

## Tasks

<task type="auto">
  <name>Create Recipients Configuration</name>
  <files>recipients.json</files>
  <action>
    Create a `recipients.json` file to store names and phone numbers. 
    - Initial content should include the current `RECIPIENT_PHONE_NUMBER` from `.env`.
    - Format: `{"recipients": [{"name": "Admin", "phone": "573002149494"}]}`.
  </action>
  <verify>Test-Path recipients.json</verify>
  <done>JSON file exists with at least one valid entry.</done>
</task>

<task type="auto">
  <name>Update main.py for Broadcasting</name>
  <files>main.py</files>
  <action>
    Modify `main.py`:
    - Import `json`.
    - Load the `recipients.json` file.
    - Loop through the recipients list.
    - Inside the loop, call `client.send_template_message` for each phone number.
    - Add basic logging to show status per recipient (e.g., "Sending to Admin...").
  </action>
  <verify>.\venv\Scripts\python.exe main.py (Verify broadcast loop executes without crashing)</verify>
  <done>Orchestrator can process a list of multiple recipients.</done>
</task>

## Success Criteria
- [ ] `main.py` successfully scrapes the TRM once and sends it to all recipients in `recipients.json`.
- [ ] The broadcast logic is robust against individual delivery failures (one failure doesn't stop the loop).
